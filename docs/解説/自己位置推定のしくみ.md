# 自己位置推定のしくみ(やさしい解説)

対象読者: ロボットの自己位置推定(SLAM・AMCL等)に馴染みがない人向け。
`go2_localization`パッケージが実際に何をしているか、なぜそう作ってあるかを
専門用語をできるだけ噛み砕いて説明する。実装の詳細は
`ros2_ws/src/go2_localization/README.md`を参照。

## 1. そもそも何がしたいのか

ロボットに「今、地図上のどこにいて、どっちを向いているか」を常に答えさせたい。
これが分かって初めて、「次にどこへ進めばいいか」(経路生成・経路追従)を計算できる。

自己位置推定の答えは、突き詰めると**「地図の座標系(map)から見て、ロボットの体(base_link)は
どこに・どんな向きであるか」**という1つの変換(平行移動+回転)を求めること。ROS2では
これを「TF(座標変換)」という共通の仕組みで表現する。

## 2. なぜ1段階ではなく2段階でやるのか

自己位置推定は「歩数を数えて今どこにいるか予想する」のと「周りを見て地図と照らし合わせて
補正する」の**2段階**でできている。片方だけでは足りない。

### 2-1. 第1段階: オドメトリ(歩数を数える) = EKF

Go2は歩くたびに、関節の角度(エンコーダ)とIMU(傾き・角速度センサ)から
「さっきの姿勢から、どれだけ・どっちに動いたか」を計算できる。これは目をつぶって
一歩ずつ距離感で移動量を積み重ねていくようなもの(=デッドレコニング)。

- 良い点: 高頻度(数十Hz)で滑らかに出せる
- 悪い点: 一歩ごとの誤差が積み重なる(ドリフトする)。何百歩も歩くと、
  最初はズレていなかったはずなのに気づけばだいぶ違う場所にいることになる

このプロジェクトでは`robot_localization`パッケージの`ekf_node`(拡張カルマンフィルタ)が、
脚から出てくる速度情報とIMUの傾き・角速度を統計的に一番もっともらしい形で合成し、
「`odom`という自分だけの局所座標系の中で、ロボットが滑らかにどれだけ動いたか」を出す。
この出力が`odom → base_link`というTFになる。

### 2-2. 第2段階: 地図とのすり合わせ(周りを見て確認する) = AMCL

デッドレコニングだけだとズレていくので、定期的に「今センサに見えている周りの形」を
「あらかじめ持っている地図」と照らし合わせて、"本当はここにいるはず"と補正する必要がある。
これがAMCL(Adaptive Monte Carlo Localization)の役目。

AMCLの考え方はやや独特で、**「ロボットはここにいるかもしれない」という仮説(パーティクル)を
何百〜何千個もばらまき**、各仮説について「もしここにいるなら、センサにはこう見えるはずだ」と
シミュレートして、実際のセンサの見え方と一致度が高い仮説ほど「本当らしい」として重みを付ける。
重みの低い仮説は淘汰され、重みの高い仮説の周りに次の世代の仮説を再配置する……というのを
センサが届くたびに繰り返す(モンテカルロ=たくさんのランダムな試行で確率的に答えに近づける手法)。
仮説の群れ(パーティクル雲)が1箇所に収束してくると「そこにいる確率が高い」と言える。

AMCLは低頻度(地図とのすり合わせが済んだときだけ)で`map → odom`というTFを補正として出す。
これがさっきの`odom → base_link`と組み合わさって、最終的に

```
map → odom → base_link
```

という3段のTFが完成し、「地図から見てロボットの体は今どこにあるか」が求まる。

> **深追い注意**: 「パーティクルが実際どう重み付け・淘汰されるか」を細部まで気にし始めると、
> パーティクルフィルタの数学的な性質(サンプル数と収束の関係、リサンプリング方式による
> 分散の違い、尤度場の設計、パーティクル退化問題...)という研究レベルの沼にそのまま
> つながっている。動かす・使う分にはここまでの理解で十分で、これ以上は「気になった時に
> 個別に調べる」でよい。ちゃんと追いたくなったら Thrun/Fox/Burgard の
> `Probabilistic Robotics`(モンテカルロ位置推定の原典的な教科書)が定番の入り口。

### 2-3. まとめ図

```
脚のエンコーダ + IMU  --[EKF]-->  odom → base_link (高頻度・滑らかだがドリフトする)
顎LiDARの点群 + 地図  --[AMCL]--> map → odom       (低頻度だが絶対位置に補正)
```

役割分担: EKFは「短い時間でどれだけ動いたか」の専門家、AMCLは「今どこにいるか」の
専門家。両方合わせて初めて実用的な自己位置推定になる。

## 3. 顎LiDARが「地図と照らし合わせる目」になる

Go2の顎には3D LiDAR(レーザーで周囲の距離を測るセンサ)が付いている。AMCLが使う
`nav2_amcl`は歴史的に**2D**の地図・2Dのレーザースキャンを前提に作られているので、
3D点群をそのまま渡せない。

そこで「点群のうち、ある高さの範囲だけを取り出して輪切りにし、2Dのレーザースキャンっぽい
形に変換する」という処理(`pointcloud_to_laserscan`パッケージ)を挟んでいる。

ここで実際にハマった話が2つある(詳細は`README.md`・GitHub Issue #26参照):

1. **顎LiDARは20°下向きに傾いて付いている**(実機スペック未確定の仮値)。単純に
   「ロボットの体の高さ±5cm」を輪切りにすると、傾いているせいで遠くの壁がほとんど
   その範囲に入らず、代わりに自分の脚や近くの床ばかり拾ってしまっていた
2. 輪切りの範囲を広げて遠くの壁を拾えるようにすると、今度は色んな距離にある
   **床**まで一緒に拾ってしまう。壁と床は「高さ」だけでは区別できない
   (床は距離が変わっても大体同じ高さにあるが、遠くの壁を拾うにはその高さ帯を
   通らざるを得ないため)

最終的には「高さで輪切りにする」のをやめて、**「このレーザー光線が、もし何もない
床まで届くとしたら何m先になるはずか」を計算し、実際に反射してきた距離がそれより
かなり手前なら"何かにぶつかった"=障害物、床に届くはずの距離とほぼ同じなら"床"**、
という判定方法に変更した(`height_slice_viz`ノード)。単なる高さのしきい値ではなく、
「センサの取り付け角度から逆算した理論値との比較」にすることで、床と壁をかなり
綺麗に分離できるようになった。

## 4. このリポジトリでの実装との対応

| 概念 | 実際のノード・トピック |
|------|------------------------|
| オドメトリ+IMU融合(EKF) | `robot_localization`の`ekf_node` → `/go2_localization/odometry/filtered` |
| 3D点群→2D変換 | `pointcloud_to_laserscan` → `/go2_localization/chin_lidar_scan` |
| 床除去した点群(可視化用) | 自作`height_slice_viz`ノード → `/go2_localization/chin_lidar_scan_points` |
| 地図とのすり合わせ(AMCL) | `nav2_amcl` → `/go2_localization/amcl_pose`(推定姿勢)・`/go2_localization/particlecloud`(パーティクル雲) |
| 地図そのもの | `nav2_map_server` → `/go2_localization/map` |

このパッケージは practice用に**自分のEKF・AMCLの推定結果は`/go2_localization/`以下の
専用トピックにしか出さない**設計になっている。理由は、simコンテナ(Gazebo環境)側が
既に本家のEKF・AMCLを動かしていて、同じ名前(`map`・`odom`等)のTFを2つのノードが
同時に配信すると衝突するため。詳しくは`ros2_ws/src/go2_localization/README.md`の
「なぜこの設計になっているか」を参照。

## 5. 実際どうやって開発しているか(アルゴリズムは自作していない)

ここまでEKF・AMCLの中身を説明してきたが、誤解しないでほしいのは
**これらのアルゴリズム自体は自分たちで実装していない**ということ。実際にやったのは:

1. **既存の実装が入ったパッケージをそのまま使う**。EKFは`robot_localization`、
   AMCLは`nav2_amcl`、3D→2D変換は`pointcloud_to_laserscan`、地図配信は`nav2_map_server`——
   どれもROS2の世界で何年も使われてきた既製品(`apt install ros-humble-<パッケージ名>`で
   入る)。カルマンフィルタやパーティクルフィルタを自分で書くことは一切していない
2. Gazeboシミュレーション環境まるごとのような大きな外部資産は、**git submoduleで
   丸ごと持ってくる**(`external/go2_ros2_sim_py`)。これも中身(Gazeboプラグイン・
   ロボット記述ファイル等)は書き換えず、必要な差分だけこちら側で足す方針
3. 自分たちが実際に書いたのはほぼ2種類だけ:
   - **yamlの設定ファイル**(`config/*.yaml`): 各パッケージのGitHub README・公式ドキュメント・
     場合によってはソースコード中のパラメータ一覧を読んで、「このパラメータはこういう意味だから
     うちのロボットだとこの値」と当てはめていく作業。ゼロから考えて出した数値ではなく、
     既存パッケージの説明書通りに埋めていっただけ、というのが実態に近い
   - **launchファイル(Python)**: 上記の既製ノードを「どの順番で・どのトピック名で・
     どのTFフレームで」起動して繋ぐか、という配線図を書くだけ
   - ごく一部、既製パッケージでは対応できない処理だけ小さな自作ノードにした
     (`height_slice_viz`の床除去ロジック、`cmd_vel_safety`の速度クランプ等)。
     それも「アルゴリズムを発明する」のではなく、既知の考え方(幾何計算・レート制限)を
     数十行のコードに落とすだけの範囲にとどめている

つまり開発の実態は「**既製のアルゴリズム部品を集めてきて、それぞれの説明書通りに設定し、
配線する**」作業がほとんどで、独自に書いたコードは全体のごく一部。これは手抜きというより
意図的な方針で、AMCLやEKFを自作しても何年もかけて磨かれた既製品ほどの完成度・実績には
まず届かない。自分たちが価値を出すべきなのは「Go2固有のセンサ配置に合わせて既製品を
正しく組み合わせる部分」と「既製品が対応していない隙間を埋める小さな部品」の方。

(このリポジトリ全体での開発サイクル・ハマりどころ・コミットの型といった、より実践的な
「作業の型」は`開発ガイド.md`にまとめてあるので、そちらも参照)

## 6. 今わかっている限界・今後の課題

- **壁への衝突に弱い**: 衝突で機体が外力を受けると、脚のエンコーダは「普通に歩いた分」しか
  検知できず、衝突によるズレをEKF(延いてはAMCLの動きモデル)が認識できない。結果として
  ズレたまま戻らないことがある(自M3の頑健性評価で扱う予定)
- **ロボット自身の脚への反射**: 顎LiDARの至近距離(0.5m未満)は、環境ではなく
  ロボット自身の脚・胴体に反射している。床除去とは別の対策(最小距離のカットオフ等)が
  今のところ未実装
- **顎LiDARの搭載角度・位置は実機の裏付けがない仮値**: 実機の型番・搭載位置が
  確定次第、幾何計算のもとになる数値ごと見直しが必要
