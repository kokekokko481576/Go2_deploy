# 解説：外部サブモジュールの使い方（submodule / fork とは）

このプロジェクトは3つの外部リポジトリを `git submodule` として取り込んでいるが、
それぞれ使い方（fork するかどうか、改変するかどうか）が異なる。混乱しやすいので、
「submodule とは何か」「fork とは何か」「変更がどう反映されるか」を整理する。

## 1. このプロジェクトが使っている3つのsubmodule

| path | 参照先URL | 種別 | このプロジェクトでの改変 |
|---|---|---|---|
| `chapter1` | `AI-Robot-Book-Humble/chapter1`(公式) | 未チェックアウト（`update=none, ignore=all`） | なし。触らない教材コード |
| `external/unitree_ros2` | `unitreerobotics/unitree_ros2`(公式そのまま) | 公式リポジトリを直接参照 | なし（バグはDockerfile側の後処理で回避、本体無改変） |
| `external/go2_ros2_sim_py` | `kokekokko481576/go2_ros2_sim_py`(**自分のfork**) | fork+改変 | あり（顎3D LiDAR追加など） |

つまり「公式をそのまま使う」「公式をforkして自分の変更を入れる」「そもそも使わず放置する」の
3パターンが混在している。以下、それぞれの仕組みを説明する。

## 2. git submodule とは

`git submodule` は「あるGitリポジトリの中に、別のGitリポジトリを、特定の1コミットに固定して
埋め込む」仕組み。親リポジトリ（`Go2_deploy`）は、埋め込んだリポジトリの中身そのものを
保存しているわけではなく、`.gitmodules` に書かれたURLと、「このコミットハッシュを使う」という
1個のポインタ（gitlinkと呼ばれる）だけを記録している。

```
Go2_deploy/               ← 親リポジトリ
  .gitmodules              ← submoduleの一覧（path・url）を書いた設定ファイル
  external/go2_ros2_sim_py ← ここに「コミット 5d07147 を使う」という参照だけがある
```

そのため：

- `git clone` した直後、submoduleのディレクトリは中身が空（もしくは`chapter1`のように
  存在すらしない）。`git submodule update --init --recursive` して初めて中身が展開される。
- submodule側のリポジトリで新しいコミットをしても、親リポジトリの記録は自動更新**されない**。
  親リポジトリ側で改めて `git add external/xxx && git commit` して、
  「参照するコミットを進めた」という変更を明示的にコミットする必要がある。
- `git status` を親リポジトリで実行すると、submoduleが指しているコミットが変わったときに
  `modified: external/go2_ros2_sim_py (new commits)` のように表示される。これは
  「中身が変わった」のではなく「参照先のコミットが変わった」という意味。

## 3. fork とは

fork は GitHub上の操作で、他人（または組織）が持つリポジトリを、自分のアカウント配下に
まるごと複製すること。複製後は完全に独立したリポジトリになり、履歴・ブランチ・すべてを
自由に書き換えられる。fork した側でどれだけ変更・pushしても、**元のリポジトリ（アップストリーム）
には一切影響しない**。逆に、元リポジトリ側が更新されても、fork側には自動反映されない
（手動で取り込む必要がある。§5参照）。

fork ↔ 元リポジトリの関係は、GitHub上の「fork」ボタンによるメタデータの紐付けと、
ローカルのgit remote設定の2つで成り立っている。このプロジェクトでは
`external/go2_ros2_sim_py` がこのパターン：

```
$ cd external/go2_ros2_sim_py && git remote -v
origin    git@github.com:kokekokko481576/go2_ros2_sim_py.git  ← 自分のfork（読み書き両方）
upstream  https://github.com/abutalipovvv/go2_ros2_sim_py.git ← 元リポジトリ（通常は読み取り専用として使う）
```

- `origin` … 自分のfork。ここに向かって `git push` すれば自分の変更が反映される
- `upstream` … 元の作者のリポジトリ。ここには基本push権限がない（PRを送るなら別途）

なお `.gitmodules` に書かれているURLは常に `origin`（このプロジェクトが実際に参照する先）で、
`upstream` は「元との差分を追跡するための補助的なremote」に過ぎない。`.gitmodules`自体には
upstream情報は含まれない。

## 4. なぜforkしたのか（このプロジェクトの実例）

`docs/docker要件定義.md` §7 に記録がある通り、顎3D LiDARの追加（2026-07-12）では
xacro/launchファイル本体の編集が必要で、docker-compose のoverrideだけでは対応できなかった。
そのため：

1. GitHub上で `abutalipovvv/go2_ros2_sim_py` を自分のアカウントへfork
   （`kokekokko481576/go2_ros2_sim_py`）
2. `.gitmodules` の参照先URLを、公式URLから自分のforkのURLに変更
3. fork先で `chin_lidar_frame` リンク・`gpu_lidar`センサ・`ros_gz_bridge`の
   ブリッジ設定を追加してcommit・push

「upstreamへPRを送る意図はない」（=独自拡張として温存する）が、「upstreamを`upstream`リモート
として残してあるので、将来 元リポジトリが更新されたら差分を取り込むことは可能」という設計。

## 5. 実際にどう反映されるか（操作の流れ）

### 5-1. fork先（go2_ros2_sim_py）を自分で改変する場合

```bash
cd external/go2_ros2_sim_py
# ファイルを編集...
git add <編集したファイル>
git commit -m "..."
git push origin main          # 自分のfork(kokekokko481576/go2_ros2_sim_py)に反映

cd ../..                       # 親リポジトリ(Go2_deploy)に戻る
git status                      # external/go2_ros2_sim_py が「new commits」と表示される
git add external/go2_ros2_sim_py
git commit -m "..."             # 親リポジトリに「参照コミットを更新した」ことを記録
```

親リポジトリのコミットには、submodule内部の差分（diff）そのものは含まれない。含まれるのは
「参照するコミットハッシュが変わった」という1行だけ。実際の変更内容を見たい場合は
submodule側のリポジトリの履歴を見る必要がある。

### 5-2. 元リポジトリ（upstream）の更新を自分のforkに取り込みたい場合

```bash
cd external/go2_ros2_sim_py
git fetch upstream
git log upstream/main -5          # 何が更新されたか確認
git merge upstream/main           # 自分の変更(main)に、元の更新を取り込む
git push origin main              # 自分のforkにも反映
# 必要なら親リポジトリ側でも submodule のポインタを更新してcommit
```

### 5-3. 公式そのまま使う場合（unitree_ros2 のパターン、forkなし）

改変しないので、fork自体が不要。新しい公式版に追従したいときは：

```bash
cd external/unitree_ros2
git fetch origin
git checkout <新しいタグ or コミット>
cd ../..
git add external/unitree_ros2
git commit -m "..."               # 親リポジトリのポインタを新バージョンへ更新
```

公式側にバグを見つけても、本体（submodule）は無改変のまま、親リポジトリ側
（`docker/driver/Dockerfile` の後処理など）で回避するという方針を、
このプロジェクトでは実際に unitree_ros2_example のインストール先指定漏れに対して取っている
（`docs/docker要件定義.md` §7参照）。

### 5-4. 使っていないsubmodule（chapter1）

`.gitmodules` で `update = none`・`ignore = all` を指定しているため、
`git submodule update --init` を実行してもチェックアウトされず、`git status`でも
差分検知の対象から外れる。参照だけ残して意図的に「触らない」状態にしている
（AI-Robot-Book教材のコードで、このプロジェクト固有の改変対象ではないため）。

## 6. まとめ

- submoduleは「特定のコミットへのポインタ」を親リポジトリに埋め込む仕組みで、
  中身の変更は親リポジトリのコミットには反映されない。ポインタの更新は別途明示的に行う
- forkは元リポジトリの独立コピー。forkへの変更は元に影響せず、元の更新もforkに自動反映されない
- このプロジェクトでは「無改変で公式を使う」（unitree_ros2）と
  「forkして改変し、upstreamリモートで追従可能にしておく」（go2_ros2_sim_py）の
  2パターンを使い分けている
